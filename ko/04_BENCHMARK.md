# Cache 효율성 벤치마크 — npm vs Standalone

> **현재 결론 (v2.1.91):** 두 설치 방식은 성능이 동일합니다. npm(Node.js 패키지 관리자로 설치하는 방식)과 standalone(단독 실행 파일로 설치하는 방식) 모두 cold start(세션을 처음 시작할 때) cache read(캐시 재사용 비율)가 84.7%이고, 안정 상태에서는 97-99%를 달성합니다. v2.1.90에 존재하던 Sentinel 격차는 사라졌습니다. 편한 쪽을 사용하면 됩니다.
>
> 아래 v2.1.90 데이터는 참고용으로 남겨둡니다 — 당시 존재했던 성능 격차와 측정 방법을 보여줍니다. v2.1.91 직접 비교는 [이 파일 하단](#v2191-addendum-april-3-2026)에 있습니다.

**v2.1.90 테스트 일자:** 2026년 4월 2일 | **v2.1.91 테스트 일자:** 2026년 4월 3일

## 환경

테스트에 사용한 장비와 설정입니다.

| 파라미터 | 값 |
|-----------|-------|
| **머신** | Linux workstation (ubuntu-1, x86-64) |
| **플랜** | Max 20 ($200/mo) |
| **버전** | v2.1.90 (npm, standalone 모두) |
| **Proxy** | cc-relay (투명 패스스루, 로깅만 수행) |
| **모니터링** | `cc-relay watch` + `cc-relay stats` + `cc-relay recent` |

cc-relay는 Claude Code와 Anthropic 서버 사이에서 데이터를 그대로 전달하면서 사용량만 기록하는 도구입니다. 테스트 결과에 영향을 주지 않습니다.

### 설치 경로

npm과 standalone은 설치되는 위치와 파일 형식이 다릅니다.

| Alias | 경로 | 바이너리 타입 |
|-------|------|-------------|
| `claude-npm` | `~/.nvm/versions/node/v22.22.1/bin/claude` → `cli.js` | **Node.js** (npm 모듈 심링크) |
| `claude` | `~/.local/bin/claude` → `~/.local/share/claude/versions/2.1.90` | **ELF 64-bit** (standalone 바이너리) |

둘 다 `ANTHROPIC_BASE_URL=http://localhost:8080` (cc-relay proxy)을 통해 라우팅됨. Proxy 소스 코드에서 **요청/응답 변조 없음** 확인 — 사용량 로깅만 수행.

쉽게 말하면, npm 버전은 Node.js(JavaScript 실행 환경) 위에서 돌아가는 스크립트이고, standalone 버전은 컴퓨터가 직접 실행하는 독립 프로그램입니다. 같은 Claude Code지만 포장 방식이 다른 것입니다.

## 테스트 설계

실제 사용 상황을 재현하기 위해 다양한 시나리오를 구성했습니다. 간단한 질문부터 대규모 파일 분석까지 단계적으로 난이도를 올렸습니다.

### npm 세션 — 시나리오

| # | 시나리오 | 목적 |
|---|----------|---------|
| 1 | 간단한 인사 ("현재 시간 알려줘") | 베이스라인(기준선) — CLAUDE.md + MEMORY.md를 처음 불러올 때의 성능 |
| 2 | 단일 파일 읽기 + 요약 (markgrab README) | Read tool 1회, 최소한의 작업 |
| 3 | 멀티 파일 코드 탐색 (feedkit 진입점 + 3개 클래스) | 여러 파일 검색과 읽기, 컨텍스트(대화 맥락) 누적 |
| 4 | 이전 컨텍스트 후속 질문 (feedkit 에러 핸들링) | 이전 대화 내용을 캐시에서 재사용하는지 테스트 |
| 5 | 병렬 비교 (markgrab vs feedkit 의존성) | Sub-agent(Claude Code가 자동으로 띄우는 보조 AI) 사용 가능성, 여러 프로젝트 동시 분석 |
| 6 | 전체 프로젝트 상태 읽기 (12+ 메모리 파일) | 대량 읽기, 넓은 컨텍스트 |
| 7 | **79개 분석 보고서** — 3개 병렬 sub-agent | 최대 부하: 보조 AI 3개가 동시에 79개 파일을 읽고 전체 요약 |

### Standalone 바이너리 세션 — 시나리오

| # | 시나리오 | 목적 |
|---|----------|---------|
| A | 대규모 코드베이스 탐색 (forge 파이프라인) | 이전 세션에 없던 완전히 새로운 컨텍스트 |
| B | 멀티 프로젝트 아키텍처 비교 (markgrab vs browsegrab) | 여러 파일 읽기, 컨텍스트 점진적 증가 |

| D | 3개 프로젝트 병렬 sub-agent (embgrep vs diffgrab vs snapgrab) | 보조 AI를 최대한 많이 동시 실행 |

## v2.1.89 베이스라인 — 망가진 상태 (4월 1일, JSONL 데이터)

모니터링 도구를 설정하기 전, 세션 기록 파일(JSONL)에서 캐시 데이터를 추출했습니다. 이 세션들은 조사를 시작하게 된 심각한 사용량 낭비를 보여줍니다.

"캐시 드레인"이란, 이전 대화 내용을 재활용하지 못하고 매번 새로 보내는 현상입니다. 캐시를 못 쓰면 같은 정보를 반복 전송하게 되어 사용량이 빠르게 소모됩니다.

| 세션 | 시각 | 엔트리 | 평균 Cache Read | 최소 | 최대 | 상태 |
|---------|------|---------|---------------|-----|-----|--------|
| `64d42269` | 4/1 16:33 | 233 | **47.0%** | 8.3% | 99.8% | 드레인 시작 |
| `9100c2d2` | 4/1 18:04 | 89 | **36.1%** | 21.1% | 89.5% | **최악의 드레인 — 이것이 조사를 촉발** |

v2.1.68 (npm)로 다운그레이드한 후:

| 세션 | 시각 | 엔트리 | 평균 Cache Read | 최소 | 최대 | 상태 |
|---------|------|---------|---------------|-----|-----|--------|
| `892388f6` | 4/1 19:12 | 119 | **97.6%** | 60.9% | 99.9% | **다운그레이드 후 복구됨** |

이전 버전으로 되돌렸더니 36.1% → 97.6%로 캐시 효율이 크게 회복됐습니다. 이는 문제가 특정 버전에서 발생한 것임을 확인해줍니다. 이후 요청 단위로 데이터를 기록하기 위해 4월 1일 17:55에 Proxy 모니터링을 설치했습니다.

## 비교

### 전체 요약

두 설치 방식을 한눈에 비교할 수 있는 표입니다. 핵심은 전체 cache read %가 거의 같다는 것입니다.

| 지표 | npm (Node.js) | Standalone (ELF) |
|--------|--------------|-----------------|
| 버전 | v2.1.90 | v2.1.90 |
| 완료 시나리오 | 1-7 (79-보고서 병렬 읽기 포함) | A-D (forge, browsegrab, feedkit 5턴, 3프로젝트 병렬) |
| **사용량 소비** | 28% → 35% (**7%**) | 35% → 40% (**5%**) |
| 전체 cache read % | **86.4%** | **86.2%** |
| Sub-agent cold start cache % | **79-87%** | **47-67%** |
| Sub-agent 워밍업 후 cache % | 87-94% | **94-99%** |
| 안정 세션 cache % | 95-99.8% | **95-99.7%** |

### Sub-Agent Cache 동작 (핵심 차이)

두 설치 모두 동일한 v2.1.90이지만, sub-agent(보조 AI)가 처음 뜰 때 캐시 효율에 차이가 있었습니다. 다만 보조 AI가 warming(워밍업)되면서 격차는 빠르게 좁혀집니다.

| Sub-Agent 단계 | npm Read % | Standalone Read % | 차이 |
|-----------------|-----------|-------------------|-------|
| Cold start (첫 1-2 요청) | 63.5-80% | **14.6-47.2%** | **-16 ~ -66pp** |
| 워밍업 (3-5 요청) | 79-87% | 67-88% | -0 ~ -20pp |
| 워밍업 완료 (5+ 요청) | 87-94% | **94-99%** | **~0pp 이상** |
| 안정 메인 세션 | 97-99.8% | **95-99.7%** | **~0pp** |

체감으로 설명하면, sub-agent를 많이 쓰는 복잡한 작업에서 standalone은 처음 1-2번 요청이 좀 더 비효율적입니다. 하지만 3-5번 주고받으면 npm과 거의 같은 효율에 도달합니다.

### 핵심 발견: v2.1.90 Standalone은 v2.1.89 대비 극적으로 개선

| 지표 | v2.1.89 Standalone | v2.1.90 Standalone | v2.1.90 npm |
|--------|-------------------|-------------------|-------------|
| Sub-agent cold start | **4-34%** | **14-47%** | **63-87%** |
| Sub-agent 워밍업 후 | N/A (복구 안 됨) | **94-99%** | **87-94%** |
| 안정 세션 | 90-99% | **95-99.7%** | 95-99.8% |

**v2.1.90 standalone은 대폭 개선됐습니다.** v2.1.89에서는 캐시가 한번 망가지면 끝까지 복구가 안 됐습니다. v2.1.90에서는 처음 1-2번만 비효율적이고 금방 정상으로 돌아옵니다. Sentinel 버그(캐시 식별 코드가 손상되는 문제)의 영향이 이제는 보조 AI 첫 시작 순간으로 제한됩니다. 워밍업 후 standalone은 **94-99% cache read를 달성합니다 — npm의 워밍업 단계와 동등하거나 오히려 더 낫습니다.**

### Token 비용 영향 (전체 데이터 업데이트)

사용량이 곧 비용입니다. 캐시를 잘 쓸수록 같은 작업에 사용량을 덜 소모합니다.

두 세션의 총 사용량: **28% → 40% = Max 20 플랜의 12%** 소비, 다음 항목을 위해:
- 7개 npm 시나리오 (3개 병렬 에이전트를 통한 79개 분석 보고서 포함)
- 4개 standalone 시나리오 (forge 탐색, 아키텍처 비교, 5턴 순차, 3프로젝트 병렬 에이전트)

| 지표 | npm | Standalone (v2.1.90) | Standalone (v2.1.89) |
|--------|-----|---------------------|---------------------|
| Sub-agent cold start 비용 | 낮음 | 보통 (첫 1-2 요청에서 2-3배) | 심각 (10-20배 지속) |
| Sub-agent 워밍업 후 비용 | ~1x | **~1x** | N/A (워밍업 안 됨) |
| 실효 세션 비용 | 1x | **~1.1-1.3x** | **~3-4x** |
| 실제 영향 | 최적 | **모든 워크플로에 수용 가능** | 사용 금지 |

v2.1.90 standalone의 추가 비용은 이제 미미합니다 — 캐시가 워밍업되기 전 첫 1-2개 sub-agent 요청으로 제한됩니다. 일상적 사용에서 npm과 standalone v2.1.90의 차이는 체감하기 어렵습니다.

### 운영 참고: 병렬 Agent vs 병렬 터미널

이 벤치마크는 **단일 터미널 내 병렬 에이전트**로 수행했습니다 (권장하는 방식). 이는 여러 터미널 창을 동시에 열어 각각 Claude Code를 실행하는 것과 다릅니다.

| 패턴 | Cache 동작 | 권장 사항 |
|---------|---------------|----------------|
| 단일 터미널, 병렬 에이전트 | 에이전트가 빌링 컨텍스트 공유, 메인 세션 cache 유지 | **안전** |
| 다중 터미널 동시 실행 | 각 터미널 = 독립 세션, cache 공유 없음, 병렬 쿼터 드레인 | **주의하여 사용** |

한마디로, 하나의 Claude Code 세션 안에서 보조 AI를 여러 개 띄우는 것은 괜찮습니다. 메인 세션의 캐시를 공유하기 때문입니다. 하지만 터미널 창을 여러 개 열어서 각각 Claude Code를 실행하면, 각 세션이 독립적으로 사용량을 소모해서 구독 한도가 빠르게 줄어듭니다.

## 차이의 원인

왜 npm과 standalone이 다르게 동작했는지 기술적 원인을 설명합니다.

### Sentinel 버그 (Bug 1) — v2.1.90에서 부분 완화, 완전 해결은 아님

Standalone 바이너리는 Bun이라는 JavaScript 런타임의 커스텀 버전으로 만든 실행 파일입니다. 이 커스텀 버전에는 특정 조건에서 캐시 식별 코드(prefix)를 손상시킬 수 있는 `cch=00000` sentinel 교체 메커니즘이 들어 있었습니다.

쉽게 말하면, 캐시가 "이건 아까 보낸 것과 같은 내용이다"를 판단할 때 쓰는 식별표가 깨지는 버그입니다. 식별표가 깨지면 같은 내용을 매번 새 것처럼 처리하게 됩니다.

**이 벤치마크에서 확인한 증거:**
- 동일 버전 (v2.1.90), 동일 머신, 동일 proxy, 동일 API key
- 유일한 변수: 바이너리 타입 (ELF vs Node.js)
- 메인 세션 cache는 **거의 동일** (~95-99%) — 두 설치 모두 안정적 대화에서 동일하게 작동
- Sub-agent cold start에서는 여전히 격차 존재 (47% vs 80%), 하지만 v2.1.90은 v2.1.89 대비 크게 개선 (47% vs 14%)
- Sub-agent 워밍업 후 격차 해소 (몇 요청 후 72-88%)

**해석:** Anthropic은 v2.1.90에서 Sentinel 버그를 부분적으로 고쳤습니다. 캐시 식별 코드 손상이 덜 심해져서 — 더 이상 매번 4-17%까지 떨어지지는 않습니다. 그러나 새 컨텍스트를 초기화할 때는 여전히 npm 대비 ~20-30pp(퍼센트 포인트) 낮은 성능을 보입니다. 근본적인 메커니즘은 여전히 남아있되 약화된 것입니다.

### 자동 업데이트 먼저 비활성화

**중요:** 테스트 전에 자동 업데이트를 꺼서 벤치마킹 중 버전이 바뀌는 것을 방지해야 합니다. Anthropic이 완전한 캐시 수정을 확인할 때까지 꺼둔 상태를 유지하는 것이 좋습니다. 예기치 않은 업데이트가 문제를 다시 일으킬 수 있기 때문입니다.

```jsonc
// ~/.claude/settings.json
{
  "env": {
    "DISABLE_AUTOUPDATER": "1"
  }
}
```

이 설정은 모든 Claude Code 호출 (standalone, npm, 패치)에 적용됩니다. 모두 `~/.claude/settings.json`을 읽기 때문입니다. npm 설치에는 자동 업데이트 기능이 없지만, 이 설정으로 standalone 바이너리가 고정되도록 보장합니다.

### 설치

npm 방식으로 설치하는 방법입니다. Node.js가 이미 설치되어 있어야 합니다.

```bash
# npm 글로벌 설치
npm install -g @anthropic-ai/claude-code

# Node.js 기반인지 확인
file $(which claude)
# 다음과 같이 표시되어야 함: symbolic link to .../cli.js (ELF executable이 아님)

# standalone 바이너리가 PATH 우선인 경우, alias 사용:
alias claude-npm="~/.nvm/versions/node/v22.22.1/bin/claude"
```

### 모니터링

캐시가 제대로 작동하는지 직접 확인하고 싶다면, 아래처럼 투명 proxy를 설정해서 사용량을 모니터링할 수 있습니다.

```bash
# cache 모니터링을 위한 투명 proxy 설정
# ANTHROPIC_BASE_URL (공식 환경 변수)을 사용하여 proxy를 통해 라우팅
ANTHROPIC_BASE_URL=http://localhost:8080 claude
```

## 재현

이 벤치마크를 직접 재현해보고 싶다면 아래 순서를 따릅니다.

1. npm과 standalone 버전을 모두 설치합니다 (서로 다른 경로에 공존 가능)
2. `ANTHROPIC_BASE_URL`을 사용하여 투명 로깅 proxy를 설정합니다
3. 각 설치를 통해 동일한 작업을 실행합니다
4. proxy 로그에서 `cache_creation_input_tokens` vs `cache_read_input_tokens`를 비교합니다
5. **sub-agent 요청**에 집중해야 합니다 — 차이가 가장 극적인 부분입니다

핵심 통찰: **안정적인 메인 세션 캐시는 두 설치 방식 간에 비슷합니다.** 버그는 주로 **새 컨텍스트가 만들어질 때** (보조 AI 시작, 새 세션, 압축 후) 나타납니다. 단순히 한 세션에서 대화만 주고받는 것으로는 차이를 발견하기 어렵습니다.

**발견 사항:**

1. **Standalone cold start 변동성**: v2.1.91 standalone은 전체 벤치마크에서 첫 요청에 27.8%를 기록했습니다 (이전 단일 프롬프트 테스트의 84.7%와 대비). 이는 워크스페이스 설정(CLAUDE.md, 메모리 파일, tool schema 등)에 따라 cold start 성능이 달라질 수 있음을 보여줍니다. npm은 일관되게 84.5%를 기록했습니다.

2. **v2.1.91에서 복구가 훨씬 빠릅니다**: Standalone은 27.8%에서 99.3%로 **단 한 번의 요청**만에 복구됐습니다. v2.1.90에서는 복구에 3-5번의 요청이 필요했습니다. 체감상, 처음 한 번만 조금 느리고 바로 정상 속도로 돌아오는 것입니다.

3. **Sub-agent 동작**: npm의 보조 AI는 54.1%에서 시작해서 93%+로 올라갑니다. Standalone의 보조 AI는 0% (완전히 처음부터 캐시 구축)에서 시작하지만 빠르게 91%+로 올라갑니다. 보조 AI 격차는 v2.1.91에도 남아있지만, 빠른 복구 덕분에 실질적 영향은 미미합니다.

4. **전체 효율**: 두 설치 모두 비슷한 전체 캐시 비율을 달성합니다 (88.4% vs 84.1%). 4pp 차이는 주로 standalone의 첫 시작 순간에서 발생하고, 워밍업 후에는 거의 같아집니다.

**권장 사항 업데이트:** v2.1.91에서는 **어느 설치 방식이든 무방합니다.** npm이 이론적으로 약간 유리하지만 (Sentinel 코드 경로가 없으므로) 실질적 차이는 이제 무시할 수 있는 수준입니다. 전체 요청별 데이터는 [06_TEST-RESULTS-0403.md](06_TEST-RESULTS-0403.md)를 참조합니다.
