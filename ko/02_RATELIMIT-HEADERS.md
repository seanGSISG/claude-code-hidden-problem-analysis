# Rate Limit 헤더 분석 — 이중 시간창 할당량 구조

> **날짜:** 2026년 4월 6일 — 48시간 프록시 관측 (4월 4~6일), v2.1.91, Max 20x ($200/mo)
>
> **관련 코멘트:** [#38335](https://github.com/anthropics/claude-code/issues/38335#issuecomment-4189807108), [#41506](https://github.com/anthropics/claude-code/issues/41506#issuecomment-4189847482)

---

## 요약

Anthropic의 API는 모든 응답에 rate limit 헤더(서버가 응답에 담아 보내는 사용량 정보)를 포함합니다. Claude Code는 이 중 `representative-claim`이라는 값 하나만 읽어서 사용량 바를 표시하고, 나머지 정보는 전부 버립니다. 우리가 운영한 cc-relay 프록시(Claude Code와 서버 사이에 끼워 넣은 중계 서버)는 이 헤더를 모두 기록했습니다. 4월 4~6일 동안 헤더가 포함된 3,702건의 요청을 분석한 결과, **듀얼 슬라이딩 윈도우(일정 시간 구간을 기준으로 사용량을 계산하는 방식이 두 개 겹쳐 있는 구조)** — 5시간 윈도우와 7일 윈도우 — 가 존재한다는 것을 확인했습니다. 그리고 사용률 1%를 쓸 때마다 토큰이 얼마나 소모되는지도 측정했습니다.

**주요 발견:**
- 5시간 윈도우가 항상 병목입니다. 모든 요청의 100%에서 `representative-claim` = `five_hour`였습니다. 즉, 사용량 바는 항상 5시간 기준으로 표시됩니다.
- 5시간 사용률(utilization) 1%당 약 150만~210만 개의 "눈에 보이는 토큰"이 소비됩니다. 그중 96~99%가 cache_read(이전 대화 맥락을 다시 읽는 토큰)입니다.
- 사용자가 실제로 받는 AI 응답 토큰은 1%당 겨우 9천~1만 6천 개에 불과합니다. 이는 thinking 토큰(AI가 답변하기 전에 내부적으로 생각하는 과정에서 소모되는 토큰)이나 cache-read 가중치, 또는 둘 다가 할당량의 대부분을 차지한다는 뜻입니다.
- 7일 카운터는 5시간 사용량에 비례해서 쌓입니다 (비율 약 0.12~0.17).
- 5시간 윈도우는 대략 5시간마다 초기화됩니다. 7일 윈도우는 매주 정해진 시각에 초기화됩니다.

실감나는 수치로 바꿔 보면: **한 달 $200 구독에서, 5시간 안에 사용량의 39%가 소진된 경우가 실제로 관측됐습니다.** 하루에 몇 번만 집중 작업해도 금방 제한에 걸릴 수 있다는 뜻입니다.

## 1. 헤더 필드

모든 API 응답에 다음 헤더가 포함됩니다. 이 값들이 사용량이 얼마나 찼는지를 서버가 알려주는 정보입니다.

```
anthropic-ratelimit-unified-5h-utilization: 0.26
anthropic-ratelimit-unified-5h-reset: 1775455200
anthropic-ratelimit-unified-7d-utilization: 0.19
anthropic-ratelimit-unified-7d-reset: 1775973600
anthropic-ratelimit-unified-representative-claim: five_hour
anthropic-ratelimit-unified-fallback-percentage: 0.5
anthropic-ratelimit-unified-overage-status: allowed
anthropic-ratelimit-unified-overage-utilization: 0.0
```

| 헤더 | 설명 |
|--------|-------------|
| `5h-utilization` | 현재 5시간 윈도우의 사용률 (0.0 = 비어 있음, 1.0 = 전부 소진) |
| `5h-reset` | 이 5시간 윈도우가 끝나고 초기화되는 시각 (Unix 타임스탬프) |
| `7d-utilization` | 현재 7일 윈도우의 사용률 |
| `7d-reset` | 이 7일 윈도우가 끝나고 초기화되는 시각 |
| `representative-claim` | 5시간과 7일 중 어느 쪽이 더 빡빡한 제약인지를 나타냅니다. Claude Code 화면에 표시되는 값입니다 |
| `fallback-percentage` | 용량 할당 비율 (일관되게 0.5) |
| `overage-status` | 초과 사용 과금 상태 |
| `overage-utilization` | 초과 사용 예산 중 소비된 양 |

### 종합 관측 (n=3,702)

- `representative-claim` = `five_hour` — 요청의 **100%**. 관측 기간 내내 7일 윈도우가 병목이 된 적은 한 번도 없었습니다.
- `fallback-percentage` = `0.5` — 요청의 100%
- `overage-utilization` = `0.0` — 전 기간 동안
- 5시간 리셋 타임스탬프 관측값: 14:00, 19:00, 01:00, 12:00, 18:00, 23:00, 04:00, 09:00 KST — 대부분 5시간 간격이지만 두 가지 예외가 있었습니다. 19:00→01:00은 6시간, 01:00→12:00은 11시간이었는데, 사용하지 않는 시간대에 관측되지 않은 윈도우 경계가 있었을 가능성이 있습니다. 정확한 패턴을 확인하려면 추가 데이터가 필요합니다.
- 7일 리셋: 4월 10일 12:00 KST (고정 주간 주기, 모든 응답에서 일관됨)

## 3. 5시간 사용률 1%당 비용

총 토큰을 사용률 퍼센트로 나눈 값입니다. 쉽게 말해, "사용량 바가 1% 올라갈 때 토큰이 얼마나 소모됐나"를 계산한 것입니다.

"가시 토큰 합계"는 Output + Cache Read + Input을 모두 합친 값입니다. thinking 토큰은 API 응답에 포함되지 않기 때문에 측정이 불가능해서 제외했습니다.

| 윈도우 | 최대값 | 1%당 Output | 1%당 Cache Read | 1%당 가시 토큰 합계 |
|--------|------|---------------|-------------------|---------------------|
| A | 0.39 | 15,840 | 1,779,834 | 1,849,471 |
| C | 0.27 | 13,341 | 2,057,274 | 2,091,707 |
| E | 0.26 | 11,523 | 1,746,063 | 1,778,562 |
| F | 0.21 | 9,163 | 1,524,058 | 1,554,070 |
| G | 0.18 | 10,560 | 1,893,532 | 1,910,357 |

핵심 포인트:
- Cache_read가 사용률 1%당 눈에 보이는 토큰의 96~99%를 차지합니다. 대부분의 토큰이 "이전 대화를 다시 읽는 데" 소비되고 있다는 뜻입니다.
- 1%당 실제 AI 답변(Output): 9K~16K (분산 약 1.73배)
- 1%당 Cache Read: 1.5M~2.1M (분산 약 1.35배)
- 5시간 윈도우 전체(100%)를 쓴다고 해도, 실제 AI 답변은 약 90만~160만 토큰밖에 안 됩니다.

### 사각지대: thinking 토큰

Extended thinking(AI의 확장 사고) 토큰은 API의 `output_tokens`에 포함되지 않습니다. 사용자에게는 보이지 않지만 할당량은 소모시킬 수 있습니다. 실제 1%당 비용은 다음과 같이 구성됩니다.

```
Visible output:    9K–16K       (measured)
Thinking output:   unknown      (not in API response)
Cache read:        1.5M–2.1M   (measured)
Input:             ~25K         (measured)
```

사용자 입장에서는 cache-read 가중치와 thinking 토큰이 각각 사용량에 얼마나 기여하는지 분리할 방법이 없습니다. 이것이 가장 큰 불투명 지점입니다.

## 4. 7일 누적 패턴

5시간 윈도우의 사용량이 7일 윈도우에 어떻게 쌓이는지를 보여주는 데이터입니다. 7일 변화량이 0인 윈도우 B와 H는 제외했습니다. 참고로 윈도우 D(최대 0.08)는 섹션 3의 1%당 테이블에서는 제외됐지만 여기에는 포함했습니다. 7일 누적은 적은 사용량에서도 관측 가능하지만, 1%당 비용 계산은 사용률이 너무 낮으면 정확도가 떨어지기 때문입니다.

| 윈도우 | 5h 최대값 | 7d 변화량 | 비율 (7d/5h) |
|--------|---------|----------|---------------|
| A | 0.39 | +0.05 | 0.128 |
| C | 0.27 | +0.04 | 0.148 |
| D | 0.08 | +0.01 | 0.125 |
| E | 0.26 | +0.03 | 0.115 |
| F | 0.21 | +0.03 | 0.143 |
| G | 0.18 | +0.03 | 0.167 |

비율이 0.12~0.17 사이에 모여 있습니다. 방향성은 일관되지만 정밀하지는 않습니다. 7일 값은 소수점 둘째 자리까지만 표시되기 때문에, +0.03은 실제로 0.025~0.035 사이일 수 있습니다. 데이터 포인트가 6개뿐이라 확정적인 공식을 뽑기에는 부족합니다.

쉽게 말하면: **5시간 윈도우에서 사용량의 약 12~17%가 7일 윈도우에도 누적됩니다.** 매일 열심히 쓰면 7일 윈도우도 결국 차게 됩니다.

## 5. v2.1.89 버그와의 구분

v2.1.89 캐시 회귀 버그(3월 28일 ~ 4월 1일)는 용량 축소와는 **별개의, 이미 해결된** 문제입니다. 혼동하지 않도록 시기별로 정리했습니다.

| 기간 | 캐시 상태 | 성격 |
|--------|-------------|--------|
| 3/23~27 ("골든 기간") | 98~99% | 정상 상태, 비교 기준 |
| 3/28~4/1 (v2.1.89 버그) | 86%, 29% 깨짐 | 캐시 버그. 용량 분석에서 제외 |
| 4/2~5 (v2.1.91+) | 84~97%, 0% 깨짐 | 버그 수정 후 — 그런데도 용량 축소가 관측됨 |

이 문서의 모든 프록시 데이터는 버그 수정 후 기간의 것입니다. 즉, **캐시 버그가 고쳐진 이후에도 사용량이 빨리 소진되는 현상은 따로 존재한다**는 뜻입니다.

### 우리 분석과의 관계

이 두 분석은 **사용자 쪽에서 주고받은 토큰 양**을 측정한 것입니다. 우리 프록시 데이터는 **서버가 그 사용량을 어떻게 점수로 매겼는지**를 추가로 보여줍니다. 어느 한쪽만으로는 전체 그림을 볼 수 없고, 양쪽을 합쳐야 "왜 이렇게 빨리 소진되는가"의 방정식 양변을 모두 볼 수 있습니다.
