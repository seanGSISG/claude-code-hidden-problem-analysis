# Rate Limit 헤더 분석 — 이중 시간창 할당량 구조

> **날짜:** 2026년 4월 6일 — 48시간 프록시 관측 (4월 4~6일), v2.1.91, Max 20x ($200/mo)
>
> **관련 코멘트:** [#38335](https://github.com/anthropics/claude-code/issues/38335#issuecomment-4189807108), [#41506](https://github.com/anthropics/claude-code/issues/41506#issuecomment-4189847482)

---

## 요약

Anthropic의 API는 모든 응답에 `anthropic-ratelimit-unified-*` 헤더를 포함합니다. Claude Code는 이 중 `representative-claim` 하나만 읽어서 사용량 바를 표시하고, 나머지는 전부 버립니다. 우리가 운영한 cc-relay 프록시는 이 헤더를 모두 기록했습니다. 4월 4~22일 동안 헤더가 포함된 **37,363건**의 요청을 분석한 결과, **듀얼 슬라이딩 윈도우** 시스템 — 5시간 윈도우와 7일 윈도우 — 가 존재한다는 것을 확인했으며, 사용률 1%당 토큰 소모량도 측정했습니다.

**주요 발견:**
- 5시간 윈도우가 요청의 **77.4%**에서 병목입니다 (`representative-claim` = `five_hour`). **7일 윈도우도 제약 조건이 될 수 있습니다** — 4월 9~10일에 7일 사용률이 0.85~0.97에 도달했을 때 관측되었습니다 ([9절](#9-seven_day-병목--첫-관측-4월-914일) 참조)
- 5시간 사용률 1%당 약 150만~210만 개의 가시 토큰이 소비되며, 그중 96~99%가 cache_read입니다
- 가시 output 토큰은 1%당 겨우 9K~16K에 불과합니다 — thinking 토큰이나 cache-read 가중치(또는 둘 다)가 할당량의 대부분을 차지한다는 것을 시사합니다
- 7일 카운터는 5시간 사용량에 비례해서 쌓입니다 (비율 약 0.12~0.17)
- 5시간 윈도우는 대략 5시간 간격으로 초기화됩니다. 7일 윈도우는 매주 고정된 시각에 초기화됩니다

## 1. 헤더 필드

모든 API 응답에 다음 헤더가 포함됩니다:

```
anthropic-ratelimit-unified-5h-utilization: 0.26
anthropic-ratelimit-unified-5h-reset: 1775455200
anthropic-ratelimit-unified-7d-utilization: 0.19
anthropic-ratelimit-unified-7d-reset: 1775973600
anthropic-ratelimit-unified-representative-claim: five_hour
anthropic-ratelimit-unified-fallback-percentage: 0.5
anthropic-ratelimit-unified-overage-status: allowed
anthropic-ratelimit-unified-overage-utilization: 0.0
```

| 헤더 | 설명 |
|--------|-------------|
| `5h-utilization` | 현재 5시간 윈도우의 사용률 (0.0 = 비어 있음, 1.0 = 전부 소진) |
| `5h-reset` | 이 5시간 윈도우가 끝나고 초기화되는 시각 (Unix 타임스탬프) |
| `7d-utilization` | 현재 7일 윈도우의 사용률 |
| `7d-reset` | 이 7일 윈도우가 끝나고 초기화되는 시각 |
| `representative-claim` | 어느 윈도우가 더 빡빡한 제약인지 — CLI에 표시되는 값입니다 |
| `fallback-percentage` | 용량 할당 비율 (일관되게 0.5) |
| `overage-status` | 초과 사용 과금 상태 |
| `overage-utilization` | 초과 사용 예산 중 소비된 양 |

### 종합 관측 (초기 n=3,702; 4월 14일 기준 n=23,374; 4월 22일 기준 n=37,363)

- `representative-claim` = `five_hour` — 요청의 **77.4%** (18,095/23,374); `seven_day`는 **22.6%** (5,279/23,374). 초기 3,702건 샘플(4월 4~6일)에서는 100% five_hour였습니다 — 7일 병목은 7일 사용률이 용량에 근접했을 때(4월 9~10일)에야 처음 나타났습니다. [9절](#9-seven_day-병목--첫-관측-4월-914일)을 참조하십시오.
- `fallback-percentage` = `0.5` — 요청의 100% (23,374/23,374 — 14일간 분산 없음)
- `overage-status` = `allowed` — 요청의 100%; `overage-utilization` = `0.0` — 전 기간 동안
- 5시간 리셋 타임스탬프 관측값: 14:00, 19:00, 01:00, 12:00, 18:00, 23:00, 04:00, 09:00 KST — 대부분 5시간 간격이지만 두 가지 예외가 있었습니다: 19:00→01:00은 6시간, 01:00→12:00은 11시간(비활동 시간대에 관측되지 않은 윈도우 경계가 있었을 가능성이 있습니다). 정확한 경계 패턴은 추가 데이터가 필요합니다.
- 7일 리셋: 4월 10일 12:00 KST (고정 주간 주기, 모든 응답에서 일관됨)
- 관측된 최대 5시간 사용률: **0.92**; 최대 7일: **0.99** (4월 10일)

## 3. 5시간 사용률 1%당 비용

총 토큰을 사용률 퍼센트 포인트로 나눈 값입니다. "가시 토큰 합계" = Output + Cache Read + Input (측정 가능한 모든 토큰 유형; thinking 토큰은 API 응답에 포함되지 않으므로 제외):

| 윈도우 | 최대값 | 1%당 Output | 1%당 Cache Read | 1%당 가시 토큰 합계 |
|--------|------|---------------|-------------------|---------------------|
| A | 0.39 | 15,840 | 1,779,834 | 1,849,471 |
| C | 0.27 | 13,341 | 2,057,274 | 2,091,707 |
| E | 0.26 | 11,523 | 1,746,063 | 1,778,562 |
| F | 0.21 | 9,163 | 1,524,058 | 1,554,070 |
| G | 0.18 | 10,560 | 1,893,532 | 1,910,357 |

- Cache_read가 사용률 1%당 가시 토큰의 96~99%를 차지합니다
- 1%당 Output: 9K~16K (분산 약 1.73배)
- 1%당 Cache Read: 1.5M~2.1M (분산 약 1.35배)
- 5시간 윈도우 전체(100%)는 약 90만~160만 가시 output 토큰에 해당합니다

### 사각지대: thinking 토큰

Extended thinking 토큰은 API의 `output_tokens`에 포함되지 않습니다. 실제 1%당 비용은 다음과 같습니다:

```
Visible output:    9K–16K       (measured)
Thinking output:   unknown      (not in API response)
Cache read:        1.5M–2.1M   (measured)
Input:             ~25K         (measured)
```

클라이언트 측에서는 cache-read 가중치와 thinking 토큰의 기여분을 분리할 수 없습니다.

## 4. 7일 누적 패턴

7일 변화량이 측정 가능한 윈도우만 포함했습니다 (7일 변화량이 0.00인 B, H 제외). 참고: 윈도우 D(최대 0.08)는 섹션 3의 1%당 테이블에서는 제외됐지만 여기에는 포함되어 있습니다 — 7일 누적은 가벼운 윈도우에서도 측정 가능하지만, 1%당 비용 계산은 최대값이 매우 낮으면 신뢰성이 떨어지기 때문입니다.

| 윈도우 | 5h 최대값 | 7d 변화량 | 비율 (7d/5h) |
|--------|---------|----------|---------------|
| A | 0.39 | +0.05 | 0.128 |
| C | 0.27 | +0.04 | 0.148 |
| D | 0.08 | +0.01 | 0.125 |
| E | 0.26 | +0.03 | 0.115 |
| F | 0.21 | +0.03 | 0.143 |
| G | 0.18 | +0.03 | 0.167 |

비율이 0.12~0.17 사이에 모여 있습니다. 방향성은 일관되지만 정밀하지는 않습니다 — 7일 값은 소수점 둘째 자리까지만 표시되므로, +0.03은 실제로 0.025~0.035 사이일 수 있습니다. 데이터 포인트가 6개뿐이므로 확정적인 공식을 도출하기에는 부족합니다.

### 우리 분석과의 관계

두 분석 모두 **클라이언트 측 토큰 소비량**(주고받은 양)을 측정한 것입니다. 우리 프록시 데이터는 **서버 측 사용률 응답**(서버가 해당 소비량을 어떻게 점수 매겼는지)을 추가합니다. 어느 한쪽만으로는 전체 그림을 볼 수 없습니다 — 양쪽을 합쳐야 방정식의 양변을 모두 볼 수 있습니다.

### 확인된 사항

1. **계정 내 일관성:** 우리의 38,996건(16일간, 데이터셋 `ubuntu-1-stock`, 4월 1–16일) 및 cnighswonger의 11,502건(7일간) 모두 분산이 없습니다. 이 필드는 요청별, 시간대별, 부하별로 동적 조정되지 않는 **계정별 고정 값**입니다.
2. **시간에 따른 변화:** 2025년 11월(#12829)의 0.2 vs 2026년 4월의 0.5 — 값은 시간이 지나면서 변경될 수 있습니다.
3. **같은 플랜에서도 `overage-status`가 다릅니다:** 우리의 Max 20x와 cnighswonger의 Max 5x는 모두 `allowed`이지만, 0xNightDev의 Max 5x는 `rejected` + `org_level_disabled`입니다.

### 미확인 사항

`fallback-percentage`의 **정확한 의미**는 Anthropic에서 문서화하지 않았습니다. 커뮤니티 해석에는 다음이 포함됩니다: 승수형 할당량 상한(0.5 = 이론적 최대의 50%), 폴백 라우팅 비율, 또는 소진 후 성능 저하 임계값. [@cnighswonger는 다음과 같이 경고했습니다](https://github.com/anthropics/claude-code/issues/41930#issuecomment-4230013175): *"Your three interpretations are all defensible and we can't distinguish them from the data we have."*

**우리는 관측된 값을 문서화하되 어떤 해석도 지지하지 않습니다.** 공식 문서가 없는 상태에서 특정 해석을 사실로 취급하면 이 저장소의 증거 기준을 훼손하게 됩니다. 4월 13일 기준 #41930에서 이 필드에 대한 Anthropic의 응답은 없었습니다.

## 9. `seven_day` 병목 — 첫 관측 (4월 9~14일)

> **추가:** 2026년 4월 14일

섹션 1~7에서는 초기 3,702건(4월 4~6일)에서 `representative-claim` = `five_hour`가 100%였다고 보고했습니다. 확장된 데이터셋(23,374건, 4월 4~14일)에서 7일 윈도우가 처음으로 제약 조건으로 관측되었습니다:

| 날짜 | `five_hour` | `seven_day` | 평균 5h util | 평균 7d util |
|-----|-------------|-------------|-------------|-------------|
| 4/4~8 | 12,072 | 0 | 0.10~0.30 | 0.10~0.63 |
| **4/9** | **141** | **4,148** | 0.037 | **0.848** |
| **4/10** | 1,366 | **1,131** | 0.229 | **0.966** |
| 4/11~14 | 4,518 | 0 | 0.019~0.306 | 0.195~0.442 |

**무슨 일이 있었나:** 4월 9일까지 누적 7일 사용량이 0.85에 도달했습니다(주간 상한 근접). 리셋 후 5시간 윈도우는 거의 비어 있었지만(0.037), 7일 윈도우가 더 빡빡한 제약이었기 때문에 `representative-claim`이 `seven_day`로 전환되었습니다. 4월 10일에 7일 사용률이 **0.966**(거의 소진)으로 정점을 찍었습니다. 주간 리셋(4월 10일 12:00 KST) 후 7일 사용률이 하락하고 `five_hour`가 다시 병목으로 복귀했습니다.

**시사점:**
1. 7일 윈도우는 **단순히 표시용이 아닙니다** — 지속적인 과다 사용 시 실질적인 rate limiter가 될 수 있습니다
2. 일주일 내내 CC를 집중적으로 사용하는 사용자(단발적 사용이 아닌)는 5시간 상한보다 먼저 7일 상한에 도달할 수 있습니다
3. 5시간 사용률 분포가 변합니다: 7일이 제약일 때 5시간은 활발한 작업 중에도 낮게 유지되며 — 표시되는 사용량 바가 실제 제약을 과소평가할 수 있습니다

**수정된 발견:** 5시간 윈도우가 요청의 **77.4%** (18,095/23,374)에서 병목이며, 이전에 보고한 100%가 아닙니다. 7일 윈도우가 **22.6%** (5,279/23,374)에서 병목이었고, 4월 9~10일에 집중되었습니다.

## 10. 데이터 해석 주의사항 — 환경 변경 (4월 14일)

> **추가:** 2026년 4월 14일

데이터셋 `ubuntu-1-stock`의 프록시 데이터는 4월 1–16일에 걸쳐 있습니다(**총 38,996건**, 4월 14일 분석 시점까지 rate limit 헤더 포함 23,374건). 이 기간 동안 측정 환경이 변경되었습니다:

| 기간 | 요청 수 | 환경 | 용도 |
|--------|----------|-------------|---------|
| 4/1 ~ 4/10 14:25 | 25,558 | **미수정** CC — flag 오버라이드 없음 | 기준선 측정, 버그 이벤트 카운트, 1%당 비용 분석 |
| 4/10 14:25 ~ 4/15 | 9,996 | GrowthBook flag 오버라이드 활성 (B4/B5 flag를 허용값으로 설정, [01_BUGS.md](01_BUGS.md#growthbook-flag-override--controlled-elimination-test-april-1014) 참조) | 오버라이드 효과 측정 전용 |

**이것이 중요한 이유:**
- B4/B5 이벤트 카운트(167,818 및 5,500)는 전부 미수정 기간의 데이터입니다
- Rate limit 헤더 분석(1~7절)은 4월 4~6일 데이터(미수정 기간)를 사용하며 영향을 받지 않습니다
- `seven_day` 병목 관측(9절)은 양쪽 기간에 걸쳐 있지만, 할당량 사용률 수준에 의해 결정되며 flag 오버라이드에 의한 것이 아닙니다 — flag 오버라이드는 context 변조에 영향을 미치지 할당량 산정에는 영향을 미치지 않습니다
- `fallback-percentage` 데이터(8절)는 양쪽 기간에 걸쳐 있으며, 양쪽 모두 0.5입니다. 이는 오버라이드가 이 필드에 영향을 미치지 않음을 확인합니다

이 저장소의 데이터를 인용할 때는 올바른 환경 맥락을 확보하기 위해 측정 기간에 유의하십시오.
