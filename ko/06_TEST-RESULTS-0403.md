## 버그 검증 결과

### B3: Synthetic Rate Limiter (가짜 속도 제한) — 확인됨 (151건)

**방법:** 기존 JSONL (로그 파일) 전체에서 `<synthetic>` (가짜 응답) model 항목을 검색했습니다.

**결과:**
- **65개 세션 파일**에 걸쳐 **151개의 `<synthetic>` 항목**을 발견했습니다
- 25개는 `acompact` sub-agent (자동 압축 보조 작업)에서, 126개는 일반 세션에서 나왔습니다
- 시그니처 (고유 패턴): `model=<synthetic> input=0 output=0 stop=stop_sequence`
- 표시되는 텍스트: `"No response requested."`

**해석:** Claude Code가 실제로 AI에게 질문하지 않고도, 클라이언트 측 rate limiter (사용 속도를 제한하는 장치)가 작동합니다. 우리 환경(v2.1.90)에서 직접 확인했습니다. 드물게 발생하는 문제가 아닙니다 — 65개 세션이 영향을 받았습니다.

### B4: Microcompact (마이크로 압축) — 확인됨 (총 327건, 패턴 분석 완료)

**방법:** cc-relay의 `_scan_microcompact()` 기능이, Claude Code가 서버로 보내는 요청 안에서 `[Old tool result content cleared]` (이전 도구 결과 내용이 삭제됨)라는 표시를 감지했습니다.

**결과 (v2.1.90, 기존 데이터 기반):**

| 세션 | 이벤트 수 | 클리어된 인덱스 | 메시지 범위 | Tool 결과 |
|------|----------|----------------|------------|----------|
| 세션 1 (02:29-02:54) | 67 | [20,22,38,40,44,46,162,166,172,174,206] | 121→251 | 53→112 |
| 세션 2 (09:23-09:47) | 71 | [2,4,6,8,10,12,58,60,68,70,74,76] | 59→135 | 38→81 |

**주요 발견사항:**

1. **클리어된 인덱스가 모두 짝수입니다** → 도구 호출과 그 결과가 한 쌍으로 묶여서 삭제 대상이 됩니다
2. **삭제 범위가 시간이 지남에 따라 넓어집니다** — 처음에는 오래된 메시지만 지우다가, 점점 최근 메시지까지 지웁니다
3. **메인 세션의 cache (캐시, 재사용 가능한 저장 데이터)는 영향 없습니다** — 삭제 중에도 캐시 적중률이 99% 이상을 유지합니다 (기존 내용의 앞부분을 그대로 유지하는 방식 덕분입니다)
4. **Sub-agent (보조 작업) 첫 시작 시에는 캐시가 떨어집니다** — 삭제 시점에 0%, 39%, 82%가 관측되었습니다
5. **모든 GrowthBook gate (기능 켜고 끄는 스위치)가 비활성화 상태**에서도 삭제가 발생합니다 → 공식적으로 알려진 3개의 스위치와는 별개인 숨겨진 압축 경로가 있다는 뜻입니다

**v2.1.91 검증:**
- v2.1.91 테스트 세션에서도 14건의 microcompact 이벤트가 발생했습니다
- 동일한 인덱스 패턴 `[58, 60, 68, 70, 74, 76]`
- **v2.1.90과 변화 없습니다** — v2.1.91에서 이 문제를 고치지 않았습니다

**Cache 영향:**

| 컨텍스트 | 클리어링 중 | 클리어링 없을 때 |
|----------|-----------|-----------------|
| 메인 세션 | **99%+** (영향 없음) | 99%+ |
| Sub-agent cold start | **0-39%** | 47-87% |
| Sub-agent warmed | **94-99%** | 94-99% |

**수정된 평가:** Microcompact는 메인 세션에서 캐시 효율을 떨어뜨리지는 않습니다. 진짜 피해는 **대화 품질 저하**입니다 — Claude가 이전에 파일을 읽거나 명령을 실행한 결과를 잃어버려서, 앞에서 한 작업을 정확히 기억하거나 인용할 수 없게 됩니다.

### B5: Tool Result Budget Enforcement (도구 결과 용량 제한) — 확인됨 (활성, 200K 상한)

**방법:** cc-relay의 `_scan_budget_enforcement()` 기능이 서버로 보내는 요청에 포함된 도구 결과의 크기를 측정했습니다.

**GrowthBook 플래그 (기능 스위치, 활성 확인됨):**
```
tengu_hawthorn_window:       200,000 (aggregate cap)
tengu_pewter_kestrel:        {global: 50000, Bash: 30000, Grep: 20000, Snip: 1000}
tengu_summarize_tool_results: True
```

**결과:**
- **242,094자** 시점에서 budget 초과가 감지되었습니다 (200K 상한 초과)
- `suspiciously_small` (의심스러울 정도로 작음) 마커가 붙은 **261건의 budget 이벤트**가 발생했습니다
- 도구 결과가 **1-41자**로 축소되었습니다 (원래는 수천 자 이상이었습니다)
- 예시: `msg[32] 1 char`, `msg[42] 2 chars`, `msg[172] 22 chars` (x33 발생)

**v2.1.91 검증:**
- v2.1.91 테스트 세션에서도 71건의 budget 이벤트가 발생했습니다
- 동일한 패턴입니다 — `maxResultSizeChars` override (크기 제한 상향 설정)는 MCP (외부 도구 연결 프로토콜) 전용이라서 built-in tool (파일 읽기, 검색 등 기본 도구)에는 적용되지 않습니다

**영향:** 월 $200 내고 쓰는 1M context (100만 토큰 대화 길이)인데, 기본 도구 결과에 사실상 **200K 상한**이 걸려 있습니다. 이 한도를 넘기면 이전 도구 결과가 사용자 모르게 잘립니다.

### B8: JSONL 로그 중복 — 확인됨 (2.87배 부풀리기)

**방법:** `tools/jsonl_analyzer.py`가 PRELIM (예비 기록)과 FINAL (최종 기록) 항목을 분류했습니다.

**세션별 결과:**

| 세션 | 크기 | PRELIM | FINAL | 비율 | 부풀리기 |
|------|------|--------|-------|------|---------|
| 현재 메인 | 1115KB | 79 | 82 | 0.96x | 2.87x |
| 이전 세션 | 118KB | 16 | 6 | 2.67x | — |
| Sub-agent | 326KB | 39 | 20 | 1.95x | — |
| Sub-agent | 186KB | 12 | 7 | 1.71x | — |

**핵심:** "부풀리기" 계수는 전체 input token (모든 항목 합산)과 FINAL 항목만의 token을 비교한 값입니다. 현재 세션 기준 **2.87배**입니다. 쉽게 말해, 실제 필요한 양의 약 3배가 기록되고 있습니다. Sub-agent (보조 작업)는 일관되게 더 높은 PRELIM 비율을 보입니다.

**미결 질문:** 서버 측 rate limiter (사용량 제한 장치)가 PRELIM 항목까지 사용량으로 치는 걸까요? 만약 그렇다면, thinking (심층 사고) 기능을 쓰는 세션은 실제 사용량의 2-3배로 과다 집계되는 셈입니다.

### Cache TTL (캐시 유효 시간, Layer 0, Luong NGUYEN 분석 기반)

**외부 분석 결과:** 13시간 방치 → 350K 토큰 캐시를 처음부터 다시 구축 = 세션 budget의 9%를 소모합니다.
**우리 데이터:** 5-26분 방치 → **96%+ 캐시 유지**. 캐시가 만료되려면 훨씬 더 오래 쉬어야 합니다.

**평가:** 캐시 유효 시간은 의도된 설계이지 버그가 아닙니다. 장시간 (몇 시간) 방치한 경우에만 영향을 미치고, 일반적인 사용 패턴에서는 문제가 되지 않습니다.

- **cc-relay SQLite DB** — 다수 세션에 걸쳐 3,500건 이상의 로깅된 요청
