# JSONL 세션 로그 분석 — 1차 데이터

> **날짜:** 2026년 4월 6일 — 메인 세션 110개 + 서브에이전트 세션 279개 (4월 1-6일), v2.1.91, Max 20x ($200/월)
>
> **다른 문서와의 관계:** [02_RATELIMIT-HEADERS.md](02_RATELIMIT-HEADERS.md)는 프록시(중계 서버)에서 수집한 서버 측 rate limit(사용량 제한) 헤더를 분석합니다. 이 문서는 클라이언트 측 JSONL 세션 로그(세션 기록 파일, `~/.claude/projects/**/*.jsonl`)를 분석합니다. 섹션 5에서 양쪽 데이터를 교차 대조합니다.

## 요약

JSONL(세션 기록 파일)은 Claude Code가 API와 주고받은 모든 내용을 기록한 파일입니다. 서버 응답 헤더를 보는 프록시와 달리, JSONL에는 PRELIM/FINAL 항목(임시/최종 기록), 턴(한 번의 질문-응답)별 토큰(AI가 처리하는 텍스트 단위) 상세 내역, 합성 rate limit(클라이언트가 만들어낸 가짜 사용량 제한) 이벤트, 모델 식별 정보가 들어 있습니다.

**주요 발견:**
- **합성 rate limiting (B3):** 6일간 24건의 `<synthetic>` 항목 확인 — Claude Code가 서버와 무관하게 스스로 만들어낸 가짜 "사용량 한도 도달" 메시지입니다
- **PRELIM 부풀리기 (B8):** PRELIM(임시 기록) 5,551건 vs FINAL(최종 기록) 6,825건 (전체 비율 0.81x, 사용량이 많은 날 최대 1.14x). PRELIM 항목의 cache_read(캐시 재사용량)는 FINAL 항목의 **0.56-0.97x** 수준 — 완전히 같지는 않지만 상당히 큰 양입니다
- **서브에이전트 오버헤드:** 서브에이전트(Claude Code가 내부적으로 띄우는 보조 AI) 279개 세션이 전체 cache_read의 17.3%를 쓰면서 전체 output(AI 응답)의 40.4%를 차지했습니다. 콜드 스타트(처음 시작 시 캐시 없이 시작하는 것) 중앙값: cache_create(새 캐시 생성) 토큰 13,358개
- **세션 성장:** 990턴 세션에서 cache_read가 24배 증가했습니다 (턴당 25K에서 595K로, 선형 +575)
- **세션 비용 범위:** 가장 저렴한 세션과 가장 비싼 세션 간 753배 차이 (가시 토큰 447K에서 336M)
- **시간 정규화 비용:** cache_read 중앙값 분당 153K, 평균 분당 227K
- **프록시와의 교차 대조:** JSONL이 프록시 대비 **1.93x**의 cache_read 토큰을 기록 — PRELIM 이중 집계가 로컬 기록을 약 2배 부풀린다는 것을 직접 확인했습니다

## 3. 시간 정규화 비용

단순히 토큰 총량만 비교하면, 작업 시간이 다른 세션이나 날짜를 비교할 때 오해가 생길 수 있습니다. 시간 기준으로 정규화한(단위 시간당으로 환산한) 지표가 더 공정한 비교를 가능하게 합니다:

| 평균 지속 시간 | 158분 (2.6시간) |
| 최소 | 1분 |
| 최대 | 1,372분 (22.9시간) |

### 분당 Cache read

| 지표 | 값 |
|--------|-------|
| 중앙값 | **153,018 토큰/분** |
| 평균 | 226,675 토큰/분 |
| 최소 | 8,397 토큰/분 |
| 최대 | 864,186 토큰/분 |

### 분당 Output

| 지표 | 값 |
|--------|-------|
| 중앙값 | **574 토큰/분** |
| 평균 | 744 토큰/분 |

**이것이 왜 중요한가:** 기간별 비용을 비교할 때, 총 토큰 소비량에는 "얼마나 오래 작업했는지"와 "분당 비용이 얼마인지"가 모두 섞여 있습니다. 예를 들어, 16시간 작업에 cache_read 2.7B(시간당 168M) vs 2시간 작업에 cache_read 123M(시간당 62M)을 비교하면 총량 차이는 22배이지만, **시간당 비용 차이는 2.7배에 불과**합니다. 시간당 지표를 쓰면 "오래 일해서 많이 쓴 것"과 "같은 시간에 실제로 더 비싸게 쓴 것"을 구분할 수 있습니다.

이 점은 커뮤니티 분석(예: @fgrosswig의 3월 30일 vs 4월 6일 비교)에서도 중요합니다. 총량끼리만 비교하면 작업 시간 차이와 시간당 비용 변화가 뒤섞여서, 예산이 절감됐다고 과대 평가할 수 있기 때문입니다.

## 4. 메인 세션 vs 서브에이전트

서브에이전트(Sub-agent)란 Claude Code가 복잡한 작업을 처리하기 위해 내부적으로 따로 띄우는 보조 AI 세션입니다. 메인 세션과의 비교는 다음과 같습니다:

| 지표 | 메인 세션 | 서브에이전트 | 서브에이전트 비중 |
|--------|--------------|-----------|----------------|
| 파일 수 | 110 | 279 | — |
| 항목 수 | 11,983 | 10,178 | — |
| Output | 3,100,588 | 2,100,541 | **40.4%** |
| Cache Read | 1,607,617,725 | 335,964,567 | **17.3%** |
| Cache Create | 50,302,333 | 82,034,105 | **62.0%** |
| Input | 564,967 | 3,477,557 | — |
| Cache Read % | 96.9% | 79.7% | — |
| 항목당 평균 cache_read | **134,158** | **33,009** | 4.1x |

### 서브에이전트 콜드 스타트 비용

콜드 스타트(cold start)란 서브에이전트가 처음 시작될 때 캐시가 전혀 없는 상태에서 시작하는 것을 말합니다. 이때 cache_create(새 캐시 생성) 비용이 발생합니다:

| 지표 | 값 |
|--------|-------|
| 샘플 수 | 서브에이전트 세션 279개 |
| 중앙값 cache_create (첫 번째 턴) | **13,358 토큰** |
| 평균 | 12,105 |
| P25 — P75 | 7,205 — 16,703 |
| 최소 — 최대 | 491 — 25,075 |

서브에이전트를 하나 만들 때마다 첫 번째 턴에 대략 **cache_create 13K 토큰**이 듭니다. 6일간 서브에이전트 세션 279개를 만들었으므로, 처음 시작하는 것만으로 약 3.7M 토큰이 사용된 셈입니다.

### 서브에이전트 워밍 곡선

워밍(warming)이란 서브에이전트가 몇 턴 지나면서 캐시가 쌓여 효율이 올라가는 과정입니다:

| 지표 | 값 |
|--------|-------|
| 샘플 수 | 9,403 항목 (턴 4 이후) |
| 중앙값 cache_read % | **92.3%** |
| 평균 | 81.7% |
| P25 — P75 | 78.1% — 97.6% |
| 최소 — 최대 | 0.0% — 100.0% |

3턴 이상 후, 서브에이전트의 cache_read는 **중앙값 92.3%**로 안정됩니다. 메인 세션(96.9%)보다 낮지만 충분히 쓸 만한 수준입니다. P25(하위 25%)의 78.1%는 일부 서브에이전트가 완전히 워밍되지 않은 상태임을 보여주는데, 이는 많은 서브에이전트가 5턴도 안 되어 종료되기 때문입니다.

**이것이 왜 중요한가:**
- 서브에이전트는 **전체 응답(output)의 40%**를 만들면서 **cache_read의 17%**만 소비합니다. 서브에이전트는 대화 맥락이 짧아서 턴당 부하가 가볍습니다(33K vs 134K).
- 서브에이전트는 **cache_create의 62%**를 차지합니다. 자주 새로 시작하기 때문입니다. 매번 생성할 때마다 약 13K 토큰이 드는 비싼 작업입니다.
- Agent tool(서브에이전트 호출)을 많이 쓰는 사용자는 cache_create에서 비용이 많이 들지만, 대신 턴당 cache_read는 절약됩니다. 즉, 서브에이전트를 많이 쓰는 방식은 "시작 비용은 높지만 대화당 유지 비용은 낮은" 패턴입니다.

| 200 | 204,247 | 8.2x | 99.1% |
| 281 | **8,237** | **0.3x (이상치)** | **3.4%** |
| 301 | 257,309 | 10.4x | 97.4% |
| 500 | 347,470 | 14.0x | 99.8% |
| 700 | 439,024 | 17.7x | 99.8% |
| 990 | 595,141 | **24.0x** | 100.0% |

**증가는 일정(선형)**합니다. 대략 **턴당 cache_read +575 토큰**씩 늘어납니다. 이전 대화 내용이 계속 쌓이기 때문입니다. 턴 990에 이르면 매 API 호출마다 약 595K 토큰의 캐시된 대화 맥락을 서버로 보냅니다. 체감으로 말하면, 대화 초반에는 가벼웠던 요청이 990턴째에는 24배 무거워진 것입니다.

**턴 281의 이상치:** cache_read가 약 222K에서 8,237로 갑자기 급감하면서 cache_create가 237,490으로 급증했습니다. 이는 세션 중간에 캐시가 통째로 재구축된 것으로, microcompact(B4, 대화 내용 압축 기능)나 서버 측 캐시 삭제(eviction)에 의해 발생했을 가능성이 있습니다. 이전 수준으로 복구되는 데 약 20턴이 걸렸습니다.

**사용량 한도에 미치는 영향:** 5시간 사용률의 1%가 약 1.5M~2.1M 가시 토큰에 해당한다면 ([02_RATELIMIT-HEADERS.md](02_RATELIMIT-HEADERS.md) 참조):
- 세션 초반 (턴 10, 턴당 캐시 약 34K): 1%당 약 44~62턴 가능
- 세션 중반 (턴 500, 턴당 캐시 약 347K): 1%당 약 4~6턴 가능
- 세션 후반 (턴 990, 턴당 캐시 약 595K): 1%당 약 2.5~3.5턴 가능

**쉽게 말해, 긴 대화 후반부에서는 100턴만 써도 5시간 한도의 약 17~40%가 소진됩니다.** 이것이 긴 세션일수록 사용량 한도에 더 빨리 걸리는 이유입니다. 사용자가 같은 속도로 작업하더라도 턴당 비용 자체가 계속 올라갑니다.

## 6. 교차 대조: JSONL vs 프록시

두 데이터 소스 모두 같은 기간(4월 4~6일)의 같은 API 호출을 대상으로 하지만, 측정하는 위치가 다릅니다:
- **JSONL(세션 기록 파일)**: Claude Code 클라이언트가 기록한 내용 — PRELIM(임시) 항목 포함
- **프록시(중계 서버)**: 서버가 HTTP 응답으로 돌려준 내용 — 최종 응답만 기록

양쪽 소스의 시간별 합계를 비교했습니다 (겹치는 33시간):

| 지표 | JSONL 합계 | 프록시 합계 | 비율 |
|--------|-------------|-------------|-------|
| Output 토큰 | 1,372,025 | 1,968,953 | **0.70x** |
| Cache Read 토큰 | 663,021,317 | 344,380,772 | **1.93x** |

### Cache Read: JSONL이 프록시의 1.93배

이 수치는 버그 8(PRELIM 부풀리기)을 직접 확인해주는 증거입니다. JSONL은 최종 기록(FINAL)뿐 아니라 임시 기록(PRELIM)도 함께 저장하는데, PRELIM에도 FINAL과 **비슷한** 수준(0.56~0.97배)의 cache_read가 기록됩니다. 이 때문에 전체 수치가 거의 2배로 부풀려집니다. 약 1.93배라는 비율은 섹션 1의 PRELIM/FINAL 분석 결과와도 정확히 맞습니다 (PRELIM cache가 FINAL의 0.56~0.97배이므로, 전체 JSONL = FINAL + PRELIM ≈ FINAL만의 1.56~1.97배, 1.93배와 일치).

### Output: JSONL이 프록시의 0.70배

JSONL이 프록시보다 적은 output 토큰을 기록했습니다. 이는 주로 시간 매칭 오차 때문입니다. 서브에이전트 JSONL 파일의 수정 시간과 프록시가 해당 요청을 기록한 시간이 달라서, 일부 항목이 비교 대상 시간 범위 밖으로 빠진 것입니다.

### JSONL을 통해 사용량을 추적하는 사용자에게 의미하는 바

- JSONL 파일에서 `cache_read_input_tokens`를 합산하면, 그 합계는 실제로 API에 전송된 것보다 **약 2배 높습니다**. PRELIM(임시) 항목이 수치를 중복시키기 때문입니다.
- 만약 Anthropic의 서버 측 사용량 제한기(rate limiter)가 PRELIM 항목을 만들어내는 것과 같은 방식으로 토큰을 집계한다면, 사용자는 cache_read에 대해 이중으로 과금될 수 있습니다. 클라이언트 쪽에서 이를 확실히 확인할 수는 없지만, 1.93배라는 비율은 이 가설과 잘 맞습니다.

### 방법론 참고: 시간 정규화

서로 다른 기간의 토큰 합계를 비교할 때는, **반드시 작업 시간(분 또는 시간) 기준으로 나눠서(정규화해서) 비교해야 합니다**. 총량끼리만 비교하면 "얼마나 오래 작업했는지"와 "분당 비용이 얼마인지"가 뒤섞입니다. 예를 들어, 16시간 x 시간당 168M = 총 2.7B vs 2시간 x 시간당 62M = 총 123M은 총량 차이가 22배이지만, 시간당 비용 차이는 2.7배에 불과합니다. 시간 정규화 지표는 섹션 3을 참고하시기 바랍니다.

## 7. 제한 사항

이 분석에는 몇 가지 한계가 있습니다:

- **JSONL 타임스탬프:** 모든 항목에 정밀한 타임스탬프(시간 기록)가 있지는 않습니다. 교차 대조에서 시간별 묶음(bucket)을 사용했기 때문에 매칭 오차가 있습니다. 개별 시간대에서 타이밍 불일치로 극단적인 비율(35배 이상)이 나타날 수 있습니다.
- **서브에이전트 귀속:** 서브에이전트 JSONL 파일은 부모 세션별로 저장됩니다. 파일 수정 시간이 실제 요청 시간을 정확히 반영하지 않을 수 있습니다.
- **thinking 토큰 비가시:** 프록시와 마찬가지로, JSONL의 `output_tokens`에는 extended thinking(AI 내부 사고 과정) 토큰이 포함되지 않습니다. 실제 서버 측 턴당 output은 기록된 것보다 높습니다.
- **6일 윈도우:** 4월 1~6일만 분석한 결과입니다. 더 긴 기간을 관찰하면 주간 패턴 같은 추가 발견이 있을 수 있습니다.
- **단일 요금제 티어:** Max 20x ($200/월) 기준입니다. 다른 요금제에서는 항목별 오버헤드 특성이 다를 수 있습니다.
- **PRELIM 감지 방법:** PRELIM 항목은 비어있거나 null인 `stop_reason`(응답 종료 사유)으로 식별했습니다. 같은 조건을 가진 다른 유형의 항목이 잘못 분류되었을 가능성이 있습니다.
