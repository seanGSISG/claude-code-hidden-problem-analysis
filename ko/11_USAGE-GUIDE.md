# Claude Code 사용 가이드 — 필수 실천 사항

> **요약:**
> - v2.1.91 이상으로 업데이트하고, 자동 업데이트를 비활성화하며, 각 작업마다 새 세션을 시작하십시오.
> - CLAUDE.md는 200줄 이하로 유지하십시오 — 모든 API 호출마다 함께 전송됩니다.
> - 한 세션에서 15-20회 파일을 읽으면 오래된 도구 결과가 자동으로 잘려나갑니다. 그 전에 새 세션을 시작하십시오.
>
> **대상:** Claude Code를 사용하는 모든 분 — 개발자와 비개발자 모두 포함합니다.
> **기준 버전:** v2.1.91 이상. 설치 및 자가 진단은 [09_QUICKSTART.md](09_QUICKSTART.md)를 참조하십시오.

## 1. 버전 및 설정

| 단계 | 수행 방법 |
|------|----------|
| **버전 확인** | 터미널에서 `claude --version`을 실행합니다. **2.1.91** 이상이어야 합니다. |
| **필요 시 업데이트** | `claude update` (standalone) 또는 `npm install -g @anthropic-ai/claude-code` (npm) |
| **자동 업데이트 비활성화** | `~/.claude/settings.json`의 `"env"` 섹션에 `"DISABLE_AUTOUPDATER": "1"`을 추가합니다 (아래 예시 참조) |
| **설치 유형 확인** | `file $(which claude)` — "symbolic link" (npm) 또는 "ELF 64-bit" (standalone)이 표시됩니다 |

**왜 버전을 고정해야 합니까?** 자동 업데이트는 회귀 버그(regression, 이전에 정상이던 기능이 다시 고장나는 현상)를 유발할 수 있습니다. v2.1.89에는 캐시(prompt cache, 이전 처리 내용을 재사용하는 기능) 버그가 있어 캐시 효율이 4-17%에 불과했습니다 (정상 수치는 95% 이상). 이는 모든 토큰이 거의 정가에 가깝게 과금되었다는 의미입니다. 버전을 고정하면 다른 사용자들이 새 릴리스를 테스트한 후 원하는 시점에 업데이트할 수 있습니다.

**설정 파일 예시** (`~/.claude/settings.json`):
```json
{
  "env": {
    "DISABLE_AUTOUPDATER": "1"
  }
}
```

**npm vs standalone:** v2.1.91 이상에서는 두 방식 모두 동일하게 잘 작동합니다. 벤치마크 결과 워밍업(warm-up) 후 94-99% 캐시 효율로 수렴합니다 (측정 데이터는 [04_BENCHMARK.md](04_BENCHMARK.md) 참조). 선호하는 방식을 사용하시면 됩니다.

## 2. 세션 관리

"세션(session)"이란 Claude Code와의 하나의 연속된 대화를 의미합니다. 터미널에서 `claude`를 실행할 때마다 새로운 세션이 시작됩니다. `/clear` 명령도 같은 터미널 안에서 새로운 컨텍스트(대화 맥락)를 시작합니다.

### 새 세션을 자주 시작하십시오

이것이 여러분이 들일 수 있는 가장 효과적인 습관입니다. 세션은 브라우저 탭과 같습니다: 새 세션은 비용이 적게 들고 빠르지만, 오래된 세션은 보이지 않는 부채 — 비대해진 컨텍스트, 잘린 도구 결과, 저하된 회상 능력 — 를 축적합니다. 모든 지표가 정상으로 보일 때도 마찬가지입니다.

**실제 사례:** 데이터 파이프라인 프로젝트를 관리하는 실제 워크플로에서, 150턴 이상 지속된 단일 세션이 캐시 비율 99%를 유지했습니다. 수치적으로는 완벽하게 정상이었습니다. 그러나 Claude는 40턴 이전에 읽었던 어떤 파일도 정확하게 참조할 수 없었습니다. 이전 발견 사항을 잘못 요약하고, 이미 "본" 파일의 핵심 세부 사항을 놓치며, 자신의 이전 분석과 모순되는 접근 방식을 제안했습니다. 지표는 모든 것이 정상이라고 말했지만, 체감은 그렇지 않았습니다.

**새 세션을 시작해야 할 때:**

| 상황 | 이유 |
|------|------|
| 다른 작업이나 프로젝트로 전환할 때 | 작업 A에서 누적된 컨텍스트는 작업 B에 불필요한 노이즈입니다 |
| 15-20회 파일을 읽은 후 | 200K 글자 총량 상한(버그 5)을 넘으면 오래된 도구 결과가 자동으로 잘립니다 |
| 도구 사용 횟수가 총 30회 이상일 때 | 더 많은 결과가 지워질수록 컨텍스트 품질이 저하됩니다 |
| Claude가 이전에 읽은 내용을 "잊은" 것 같을 때 | 이는 건망증이 아니라 자동 잘림(silent truncation) 현상일 가능성이 높습니다 |
| 압축 이벤트 발생 후 (요약 메시지가 보일 때) | 압축 후 Claude는 원본 데이터가 아닌 요약본만 보유합니다 |

### 세션 순환 패턴

각 독립적인 작업 — 버그 수정, 기능 추가, 코드 리뷰 — 을 완료한 후 세션을 닫고 새 세션을 시작하십시오. 닫기 전에 핵심 컨텍스트 2-3줄을 메모에 복사합니다. 그런 다음 새 세션의 첫 번째 메시지에 붙여넣습니다. 예시:

```
Working on src/api/auth.py — added rate limiting in validate_token().
Tests in tests/test_auth.py pass. Next: wire up the /logout endpoint.
```

몇 초밖에 걸리지 않지만, Claude에게 필요한 정보만 담긴 깨끗한 컨텍스트 윈도우를 제공합니다. 멀티 프로젝트 워크플로에서 이 패턴은 세션이 수 시간 동안 날카롭게 유지되는 것과 처음 20분 후 조용히 퇴화하는 것의 차이를 만듭니다.

### --resume과 --continue를 피하십시오

`--resume`과 `--continue` 모두 전체 대화 히스토리를 과금 대상 입력으로 다시 재생합니다. 500K 토큰(텍스트 처리 단위) 대화의 경우, 한 번의 resume이 그 모든 컨텍스트를 다시 처리하는 것과 동일한 비용을 발생시킵니다. v2.1.90+의 캐시 수정이 도움이 되지만, 근본적인 높은 비용은 그대로입니다.

**대신:** 핵심 컨텍스트(파일 경로, 결정 사항, 에러 메시지)를 새 세션의 첫 번째 메시지에 복사해서 붙여넣으십시오. resume보다 빠르고 저렴합니다.

### 압축(Compaction) 이해하기

대화가 길어지면 Claude Code는 오래된 컨텍스트를 압축하여 제한 범위 안에 유지합니다. 두 가지 유형이 있습니다:

**수동 압축** (`/compact`): 사용자가 직접 실행합니다. 사용자 지정 요약 초점을 제공할 수 있습니다 — 예를 들어, `/compact focus on the database migration we discussed`와 같이 사용합니다. 같은 세션에서 계속 작업하되 관련 없는 컨텍스트를 제거하고 싶을 때 유용합니다.

**자동 압축:** 세 가지 별도의 메커니즘이 모든 API 호출 시 자동으로 실행되며, 오래된 도구 출력을 `[Old tool result content cleared]` 같은 플레이스홀더(자리 표시) 텍스트로 교체합니다. 이 메커니즘들은 환경 변수 설정과 관계없이 실행됩니다 — `DISABLE_AUTO_COMPACT=true`로 설정해도 모든 메커니즘을 막을 수는 없습니다 (기술적 세부 사항은 [05_MICROCOMPACT.md](05_MICROCOMPACT.md) 참조).

**압축 이후:** Claude는 무엇을 논의했는지(요약을 통해)는 알지만, 정확한 코드, 정확한 에러 메시지, 정확한 줄 번호는 알지 못합니다. 압축 후 "42줄의 에러가 뭐였지?"라고 물으면, Claude는 그럴듯하게 들리는 답을 지어내거나(할루시네이션) 모른다고 인정할 것입니다 — 할루시네이션의 경우가 더 위험한데, 자신 있게 보이기 때문입니다. Claude가 이전의 특정 내용을 참조해야 한다면 해당 파일을 다시 읽어야 합니다.

### /compact를 권장하지 않는 이유

실전에서 `/compact`는 예상보다 훨씬 많은 세부사항을 유실합니다. 압축 과정에서 특정 변수명, 정확한 에러 메시지, 줄 번호, 의사결정의 근거 등이 일상적으로 누락됩니다. 압축 이후 Claude는 "무엇을 논의했는지"에 대한 모호한 감각만 남고, 작업을 정확하게 이어가는 데 필요한 정밀도는 사라집니다. 이로 인해 미묘한 문제가 발생합니다: Claude가 현실과 맞지 않는 코드를 자신 있게 참조하거나, 압축 전 대화에서 이미 배제한 접근 방식을 다시 제안하는 것입니다.

**더 나은 대안: 문서 기반 세션 인수인계.** 대화를 신뢰할 수 없는 요약으로 압축하는 대신, 세션이 끝날 때 작업 문서(PLAN 파일, CLAUDE.md, 에러 로그 등)를 명시적으로 업데이트합니다 — 전체 대화 컨텍스트를 보고 내용을 검증할 수 있는 동안에 말입니다. 그런 다음 새 세션을 시작하여 해당 문서를 읽습니다.

이 "문서 교차검증" 패턴이 효과적인 이유는 다음과 같습니다:
- **무엇을 보존할지 사용자가 직접 결정합니다.** 자동 요약기가 아니라 사용자가 어떤 세부사항이 중요한지 판단합니다.
- **업데이트가 검증됩니다.** 세션을 닫기 전에 Claude에게 실제 대화 내용과 문서 업데이트를 교차 확인하도록 요청할 수 있습니다.
- **새 세션은 깨끗하게 시작됩니다.** 새 세션은 손실이 있는 AI 요약이 아니라 사용자가 검증한 권위 있는 문서를 읽습니다.
- **지식이 세션을 넘어 축적됩니다.** 각 세션의 학습 내용이 영구적인 파일에 기록되며, 결국 압축되거나 유실될 대화에 갇히지 않습니다.

여러 프로젝트를 수개월간 운용한 결과, 이 패턴은 `/compact`보다 일관되게 더 나은 성과를 보였습니다. 업데이트된 문서를 읽고 시작하는 세션은 압축된 대화 기록에 의존하는 세션보다 더 집중적이고 정확합니다 — 문서에는 사용자가 보존하기로 선택한 내용만 들어 있고, 아무것도 몰래 누락되지 않기 때문입니다.

이 접근 방식은 [Andrej Karpathy의 "LLM Wiki" 패턴](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)과 일치합니다 — 매번 쿼리할 때마다 컨텍스트를 새로 파악하는 대신(RAG나 `/compact`처럼), 구조화된 문서를 점진적으로 유지보수하며 LLM이 이를 읽고 업데이트합니다. CLAUDE.md가 스키마(행동 지침), MEMORY.md가 인덱스(주제별 파일 포인터), PLAN 파일이 작업 로그(세션별 상태) 역할을 합니다. LLM이 "요약, 교차 참조, 정리, 기록 관리"의 반복 작업을 처리하고, 사용자는 큐레이션과 검증을 담당합니다. 이상적으로는 도구 자체가 이를 자동으로 처리해야 하지만 — 그때까지는 이 수동 워크플로우가 세션 간 컨텍스트 품질을 유지하는 가장 신뢰할 수 있는 방법입니다.

## 3. 컨텍스트 이해하기 (왜 Claude가 "멍청해지는"지)

### 컨텍스트 윈도우란 무엇입니까?

하나의 대화에서 Claude가 "볼 수 있는" 모든 것을 말합니다: 사용자의 메시지, Claude의 응답, 읽은 파일 내용, 명령 출력, 검색 결과, 시스템 지시문이 포함됩니다. Claude Code는 최대 1M 토큰을 지원하지만, 자동 잘림 메커니즘으로 인해 실제 사용 가능한 컨텍스트는 이보다 훨씬 적습니다.

### 토큰 감각 기르기

토큰(token)은 Claude가 처리하는 텍스트의 측정 단위입니다. 정확하게 세어볼 필요는 없지만, 대략적인 감각이 있으면 도움이 됩니다:

| 내용 | 대략적인 토큰 수 |
|------|-----------------|
| 영어 단어 1개 | ~1.3 토큰 |
| 한국어 글자 1자 | ~1 토큰 |
| 일반적인 소스 파일 (200줄) | 2,000-5,000 토큰 |
| 긴 CLAUDE.md (500줄) | 5,000-10,000 토큰 |
| 긴 대화 (100턴) | 300,000-600,000 토큰 |

### 왜 Claude가 갑자기 성능이 떨어지는 것 같습니까?

모델이 멍청해지는 것이 아닙니다 — 이전에 읽었던 정보에 대한 접근 권한을 말 그대로 잃어버리는 것입니다.

**일어나는 일:** 한 세션에서 약 15-20회 파일을 읽으면, 도구 결과 텍스트의 총량이 200K 글자 총량 상한을 초과합니다. 그 시점에서 오래된 도구 결과가 자동으로 최소 1-41자까지 잘립니다. Claude는 해당 파일 내용, grep 결과, 명령 출력을 더 이상 볼 수 없고, 아주 작은 스텁(stub, 잔여물)만 남습니다.

**측정된 예시:** 한 테스트 세션에서 261건의 예산 잘림 이벤트(budget truncation event)가 감지되었습니다. 원래 수천 자에 달하던 도구 결과가 한 자리 수 길이로 축소되었습니다. 도구 출력 총 242,094자에서 임계값을 넘었습니다 (자세한 내용은 [01_BUGS.md](01_BUGS.md#bug-5--tool-result-budget-enforcement-all-versions) 참조).

**실제 사례:** 3개 모듈에 걸쳐 25개 파일을 읽는 코드 리뷰 세션에서, 도구 결과 예산(tool result budget)이 약 18번째 파일 즈음에 조용히 발동했습니다. Claude는 파일별 구체적인 피드백 대신 일반적인 제안을 하기 시작했습니다 — 처음 읽었던 12개 파일의 내용을 더 이상 볼 수 없었기 때문입니다. 리뷰의 품질이 "handler.py 31줄의 가드 절과 73줄의 null 체크가 충돌합니다"에서 "함수에 에러 처리를 추가하는 것을 고려하십시오"로 바뀌었습니다. 품질 저하는 점진적이 아니라 갑작스러웠습니다.

**사용자에게 어떻게 보이는지:**
- Claude가 이전에 말한 것과 모순되는 답변을 합니다 (해당 진술의 근거가 된 데이터를 더 이상 볼 수 없기 때문)
- Claude가 이미 읽었던 파일을 다시 읽습니다 (이전 결과를 기억하지 못하기 때문)
- Claude가 이미 시도했던 접근 방식을 제안합니다 (실패했다는 것을 볼 수 없기 때문)
- Claude의 답변이 더 일반적이 되고 코드베이스에 덜 기반하게 됩니다

### 대응 방법

| 조치 | 효과 |
|------|------|
| 15-20회 파일 읽기마다 새 세션 시작 | 도구 결과 예산을 초기화합니다 |
| 필요한 부분만 읽기 | 줄 범위를 사용합니다: 전체 파일 대신 "server.py의 50-80줄을 읽어줘"라고 요청합니다 |
| 요청을 구체적으로 하기 | "auth.py 42줄의 로그인 유효성 검사를 수정해줘"는 "인증 시스템을 고쳐줘"보다 파일을 적게 읽습니다 |
| CLAUDE.md를 간결하게 유지 | 턴당 오버헤드(추가 비용)를 줄입니다 (섹션 4 참조) |

## 4. CLAUDE.md 모범 사례

### CLAUDE.md는 무엇을 합니까?

CLAUDE.md는 Claude Code가 매 대화 턴(주고받기)의 시작 시 읽는 지시 파일입니다. 브리핑 문서와 같다고 생각하시면 됩니다 — Claude는 사용자의 각 메시지를 처리하기 전에 이 파일을 봅니다. 프로젝트별 규칙, 관례, 제약 사항을 넣기에 적합한 곳입니다.

### 비용

CLAUDE.md 내용은 모든 API 호출마다 캐시된 입력으로 전송됩니다. 10,000 토큰짜리 CLAUDE.md는 턴당 약 10,000 토큰의 `cache_read` 비용이 발생합니다. 50턴 세션이라면 지시사항에만 500,000 토큰이 소비됩니다. 측정 데이터에 따르면 `cache_read`가 전체 가시적 토큰 볼륨의 88-98%를 차지하며 ([03_JSONL-ANALYSIS.md](03_JSONL-ANALYSIS.md)) — CLAUDE.md는 모든 cache read의 고정 구성 요소입니다.

### 파일 계층 구조

Claude Code는 일치하는 모든 CLAUDE.md 파일을 로드하여 연결(concatenation)합니다:

| 위치 | 범위 | 예시 |
|------|------|------|
| `~/.claude/CLAUDE.md` | 전역 — 모든 프로젝트에 적용 | 개인 코딩 선호 설정 |
| `<project-root>/CLAUDE.md` | 프로젝트별 | 프로젝트 관례, 기술 스택 |
| `<project-root>/.claude/CLAUDE.md` | 프로젝트별 (대체 위치) | 대체 위치 |

일치하는 모든 파일이 함께 로드됩니다. 전역 CLAUDE.md (500 토큰)와 프로젝트 CLAUDE.md (2,000 토큰)가 있다면, 매 턴마다 2,500 토큰의 지시사항이 전송됩니다.

### 좋은 CLAUDE.md 작성법

**이렇게 하십시오:**
- 200줄 / 2,000 토큰 이하로 유지
- 프로젝트별 규칙과 제약 사항에 집중
- 글머리 기호와 짧은 문구 사용
- 내용을 직접 넣지 말고 외부 문서를 참조
- 해당 프로젝트에서 Claude가 알아야 할 것만 포함

**이렇게 하지 마십시오:**
- API 문서나 라이브러리 가이드 전체를 붙여넣기
- 긴 코드 예시나 튜토리얼 포함
- 코드베이스에 이미 있는 정보를 중복 기재
- 일반적인 코딩 조언 추가 (Claude는 이미 코딩할 줄 압니다)
- 개인 메모, 기획 문서, 작업 목록 포함

### 예시

**좋은 예 (간결하고 실행 가능 — 멀티 프로젝트 설정을 다루는 약 30줄):**
```markdown
# Project Rules
- Python 3.12, ruff for linting, pytest for tests
- Never hardcode secrets — use os.getenv() with safe defaults
- PostgreSQL: parameterized queries only, no string concatenation
- Run tests: pytest tests/ -x
- API responses: always return {data, error, timestamp}

# Git
- Never commit CLAUDE.md or PLAN.txt
- Commit message: imperative mood, <72 chars
- git email: user@users.noreply.github.com

# Key Paths
- API entry: src/api/main.py
- Pipeline config: pipeline/config.yaml
- Error log: docs/error-log.md (read on demand, not here)

# Security
- .env patterns only. No hardcoded keys.
- DELETE/DROP queries: always confirm with user first
```

이 예시는 기술 스택, 보안 정책, git 관례, 주요 파일 위치를 30줄 이하로 제공합니다. Claude가 프로젝트에서 올바르게 작업하는 데 필요한 모든 것이 담겨 있고, 불필요한 것은 없습니다.

**나쁜 예 (자연스럽게 비대해진 — 800줄 이상):**
```markdown
# Complete Project Reference
## Architecture
[ASCII diagram of 6 services, 40 lines]
## Full API Reference
### GET /api/v1/users
Returns a list of users. Parameters: limit (int), offset (int)...
### POST /api/v1/users
Creates a new user. Body: {name: string, email: string}...
[200 more lines of API docs]
## Meeting Notes (2025-03)
- Decided to switch from MySQL to PostgreSQL
- Sarah will handle the migration script
## Current Sprint Tasks
- [ ] Fix login bug (#423)
- [ ] Add rate limiting
[150 more lines of planning notes and task lists]
```

실제로, 이와 같이 수개월에 걸쳐 자연스럽게 비대해진 설정 파일이 800줄 이상에 달한 사례가 있었습니다. 전체 API 레퍼런스, ASCII 아키텍처 다이어그램, 회의록, 작업 백로그가 포함되어 있었습니다. 모든 API 호출마다 800줄 전부가 전송되었습니다. 이를 150줄로 줄인 결과 Claude의 코드 품질에는 아무런 영향이 없었지만, 턴당 오버헤드가 6,000토큰 이상 감소했습니다 — 수개월간 매 턴마다 과금되고 있던 토큰이었습니다.

### 부속 파일: MEMORY.md와 PLAN 파일

**MEMORY.md:** 자주 변경되는 정보 — 프로젝트 상태, 현재 작업, 이전 세션의 피드백 — 에는 CLAUDE.md 대신 MEMORY.md를 사용하십시오. MEMORY.md도 매 턴마다 로드되지만, 자동 관리되어 간결하게 유지하기 쉽습니다. 핵심 실천 사항: 모든 것을 한 파일에 넣지 말고 **포인터 인덱스(pointer index)** 로 사용하십시오. 각 항목은 Claude가 필요할 때 읽을 수 있는 별도 파일에 대한 한 줄짜리 참조입니다. 이렇게 하면 턴당 비용을 낮게 유지하면서도 필요할 때 상세한 컨텍스트를 사용할 수 있습니다.

**PLAN 파일:** 기획 문서(현재 작업, 접근 방식 결정, 에러 로그)는 CLAUDE.md에 넣지 말고 PLAN.txt 같은 별도 파일에 보관하십시오. Claude가 필요할 때 읽을 수 있으며, 매 턴마다 토큰 비용이 발생하지 않습니다. 이 파일들은 버전 관리에 커밋하지 마십시오 — 끊임없이 변경되는 작업 문서이므로 git 히스토리에 남길 가치가 없습니다.

## 5. 토큰 절약 습관

작은 습관이 하루 수백 턴에 걸쳐 복리로 쌓입니다. 아래 각 항목은 오늘 바로 적용할 수 있는 구체적인 변경 사항입니다.

| 습관 | 왜 도움이 됩니까 | 적용 방법 |
|------|-----------------|----------|
| **전체 파일이 아닌 줄 범위만 읽기** | 1,000줄 파일 = ~5,000 토큰. 50-80줄 = ~200 토큰. | "server.py의 50-80줄을 읽어줘"라고 요청합니다 |
| **터미널 하나만 사용** | 여러 Claude Code 세션은 캐시를 공유하지 않습니다 — 각 세션이 별도로 캐시를 구축하고 비용을 지불합니다. | 새 세션을 시작하기 전에 사용하지 않는 Claude 세션을 닫으십시오. |
| **구체적으로 요청하기** | 모호한 요청은 Claude가 컨텍스트를 찾기 위해 여러 파일을 읽게 만듭니다. | "auth.py:42의 null 체크를 수정해줘" > "인증 시스템을 고쳐줘" |
| **간결한 CLAUDE.md** | 매 턴마다 전송됩니다. 추가 5,000 토큰 x 50턴 = 250,000 토큰 낭비. | 200줄 이하로 유지합니다. 내용을 직접 넣지 말고 문서를 참조합니다. |
| **새 작업은 새 세션에서** | 오래된 세션에는 관련 없는 컨텍스트가 있고 도구 결과가 잘려 있을 수 있습니다. | 작업을 전환할 때 닫고 다시 시작합니다. |
| **백그라운드 명령 피하기** | `/dream`과 `/insights`는 백그라운드에서 API 호출을 하여 조용히 할당량을 소모합니다. | 사용하지 마십시오. |
| **목표를 먼저 명시하기** | Claude가 행동하기 전에 계획을 세울 수 있어 불필요한 파일 읽기가 줄어듭니다. | "API 코드를 보여줘"가 아니라 "/api/users 엔드포인트에 레이트 리미팅을 추가해야 해"로 시작합니다. |
| **"코드베이스를 탐색해줘"라고 말하지 마십시오** | 수십 개의 파일 읽기를 유발하여 도구 결과 예산을 소진합니다. | 작업과 관련된 2-3개 특정 파일을 Claude에게 지정하십시오. |

**모호한 탐색의 숨겨진 비용:** 가장 눈에 띄지 않는 낭비 중 하나는 Claude에게 "코드베이스를 살펴봐" 또는 "프로젝트 구조를 파악해줘"라고 요청하는 것입니다. 한 측정된 세션에서, 모호한 "API를 탐색해줘" 요청이 8분 만에 34회의 파일 읽기를 발생시켜 전체 도구 결과 예산을 소진했습니다. 실제 작업이 시작될 때쯤에는 탐색 단계에서 읽었던 모든 파일이 이미 잘린 상태였습니다. 탐색 자체가 보이지 않게 되어버렸고, Claude는 필요한 파일을 다시 읽어야 했습니다. 최종 결과: 토큰 비용 2배, 혜택 제로.

| CLAUDE.md를 정리하지 않고 계속 추가 | 수개월에 걸쳐 자연스럽게 비대해진 지시 파일 — 규칙은 추가하지만 오래된 규칙은 제거하지 않습니다. 오래된 규칙이 누적되고, 서로 모순되며, 매 턴의 비용을 부풀립니다. | 분기별로 검토하고 정리하십시오. 한 사례에서 600줄 파일을 120줄로 줄였을 때 Claude의 효과성에는 아무런 손실이 없었습니다 — 제거된 내용 대부분이 오래되었거나 중복이었습니다. |

## 7. 자가 진단

### 빠른 상태 점검

설정이 정상인지 확인하려면 다음 세 가지를 점검하십시오:

```bash
# 1. 버전 — 2.1.91 이상이어야 합니다
claude --version

# 2. 설치 유형 — "symbolic link" (npm) 또는 "ELF 64-bit" (standalone)
file $(which claude)

# 3. 활성 터미널 — Claude Code 세션은 한 번에 하나만 실행해야 합니다
# 다른 터미널에서 claude가 실행 중인지 확인합니다
ps aux | grep -c '[c]laude'
```

### 가장 유용한 진단 방법

지표를 넘어서, 어떤 로그 파일보다 더 신뢰할 수 있는 행동적 신호가 하나 있습니다: **Claude가 같은 세션에서 이미 시도한 접근 방식을 제안하기 시작하거나, 10턴 이상 전에 읽었던 파일의 특정 줄을 인용하지 못한다면, 해당 세션은 유용한 수명을 다한 것입니다.** 이것이 컨텍스트 퇴화(context degradation)의 가장 일찍 감지할 수 있는 신호입니다. 문제를 진단하려 하지 마십시오 — 그냥 새 세션을 시작하십시오. 새 세션의 비용이 퇴화된 세션에서 계속 빙빙 도는 턴의 비용보다 훨씬 적습니다.

### 세션 로그 읽기

세션 JSONL(줄 단위 JSON 로그) 파일은 `~/.claude/projects/**/*.jsonl`에 위치합니다. 각 API 호출은 토큰 사용량을 포함하는 JSON 항목으로 기록됩니다. 주요 필드 두 가지:

| 필드 | 의미 | 바람직한 값 |
|------|------|------------|
| `cache_read_input_tokens` | 서버의 프롬프트 캐시에서 재사용된 토큰 | **높을수록 좋음** (저렴 — 할인 적용) |
| `cache_creation_input_tokens` | 새로 전송된 토큰, 새 캐시 항목 생성 | **낮을수록 좋음** (비쌈 — 정가) |

**해석 방법:**

| Cache Read 비율 | 의미 | 조치 |
|----------------|------|------|
| **> 80%** | 정상. 캐시가 의도대로 작동하고 있습니다. | 조치 불필요. |
| **40-80%** | 보통. 콜드 스타트(cold start, 첫 시작)나 세션 교체가 원인일 수 있습니다. | 세션 시작 시 정상적인 현상입니다 — 안정화되는지 확인하십시오. |
| **< 40%** | 문제. 캐시가 재사용되지 않고 있습니다. | 버전을 확인하십시오 (2.1.91 이상이어야 합니다). 여러 터미널을 사용 중인지 확인하십시오. |

**JSONL 정확도에 대한 주의:** 로컬 JSONL 로그는 PRELIM 항목 중복(버그 8)으로 인해 실제 토큰 사용량의 약 1.93배를 표시합니다. 정확한 비용 추정에 로컬 로그를 의존하지 마십시오. 비율은 세션 간 비교에 유용하지만, 절대 수치는 부풀려져 있습니다. 교차 검증 데이터는 [03_JSONL-ANALYSIS.md](03_JSONL-ANALYSIS.md)를 참조하십시오.

## 8. 알려진 이슈 빠른 참조

상세한 기술 분석은 [01_BUGS.md](01_BUGS.md)에 있습니다. 이 표는 각 버그가 일상 사용에 어떤 의미를 갖는지에 대한 실용적인 요약입니다.

| 버그 | 발생 현상 | 상태 | 대응 방법 |
|------|----------|------|----------|
| **B1** Sentinel | Standalone 바이너리가 프롬프트 캐시를 깨뜨려 95% 이상이어야 할 효율이 4-17%에 불과합니다 | **수정됨** (v2.1.91) | v2.1.91 이상으로 업데이트하십시오. |
| **B2** Resume | `--resume`이 전체 캐시 미스를 유발하여 전체 대화를 재과금합니다 | **수정됨** (v2.1.90) | `--resume`과 `--continue`를 피하십시오. 새 세션을 시작하십시오. |
| **B3** False Rate Limit | 클라이언트가 API에 연락하지 않고 가짜 "Rate limit reached" 에러를 생성합니다. 6일간 24건의 합성 이벤트(synthetic event)가 측정되었습니다. | **미수정** | 유휴 시간 직후 레이트 리미트에 걸리면 Claude Code를 재시작하십시오. |
| **B4** Microcompact | 오래된 도구 결과가 자동으로 `[Old tool result content cleared]`로 교체됩니다. 327건의 이벤트가 측정되었습니다. 캐시는 99% 이상을 유지하지만, 모델은 이전 데이터에 대한 접근을 잃습니다. | **미수정** | 정기적으로 새 세션을 시작하십시오. 이를 방지하는 환경 변수는 없습니다. |
| **B5** Budget Cap | 도구 결과의 총량이 200K 글자를 초과하면 오래된 결과가 1-41자로 잘립니다. 한 세션에서 261건의 이벤트가 측정되었습니다. | **미수정** | 15-20회 파일 읽기 후 새 세션을 시작하십시오. 필요한 줄만 읽으십시오. |
| **B8** Log Inflation | 확장 사고(extended thinking)가 JSONL 로그에 2-3배 중복 항목을 생성하여 로컬 토큰 수를 부풀립니다. JSONL은 실제 사용량의 1.93배를 표시합니다. | **미수정** | 로컬 JSONL 토큰 수를 절대값으로 신뢰하지 마십시오. 상대적 비교에만 사용하십시오. |

### 서버 측 요인

클라이언트 버그 외에도 서버 측 변경이 사용 경험에 영향을 미칩니다:

- **이중 할당량 윈도우(dual quota window):** 사용량은 5시간 슬라이딩 윈도우와 7일 슬라이딩 윈도우 두 가지로 추적됩니다. 5시간 윈도우가 거의 항상 병목입니다 ([02_RATELIMIT-HEADERS.md](02_RATELIMIT-HEADERS.md)).
- **사고 토큰(thinking tokens):** 확장 사고 토큰은 `output_tokens` API 필드에 나타나지 않지만, 할당량에는 산입되는 것으로 보입니다. 가시적 출력은 관찰된 사용률의 절반도 설명하지 못합니다.
- **조직 공유:** 같은 조직에 속한 계정은 레이트 리미트 풀을 공유합니다. 동료가 Claude Code를 많이 사용하면 여러분의 가용 할당량도 줄어듭니다.
